Track D Councils & Multi-Agent

The core anti-pattern

⚠️
The trap that catches everyone
Treating architectural complexity as evidence of intellectual seriousness. More agents do not equal more thoroughness. More agents equal more token overhead, more coordination failure surface, more information degradation across handoffs, and more places for the synthesis step to silently drop signal. If your council is impressive to describe but the output is no better than one good prompt, you've built a status object, not a tool.

The six anti-patterns below are the concrete shapes this trap takes. If you spot one in your own setup, the fix is usually deletion, not refinement.

Six specific anti-patterns

ANTI 01Sequential pipelines mislabeled as parallel agents
The shape. Five "agents" arranged so each waits for the previous one's output. The diagram looks like a swim-lane chart. The execution is sequential. You're paying multi-agent overhead for single-agent throughput because there is no parallelism to exploit.
The output of step 2 references the output of step 1. If B reads A, you have a workflow, not a council.
ANTI 02Coordinator agent that does nothing but route
The shape. A "coordinator" agent that reads a request, picks one of three branches, and dispatches. The routing logic is a five-line if/else. You pay for an LLM call (and ~20K tokens of context-loading overhead) to do what one line of code could do deterministically.
The coordinator's prompt is a list of rules. If the rules are deterministic, the coordinator should be code.
ANTI 03Tiny tasks given to specialized agents
The shape. Spinning up the accessibility-tester subagent to check one CSS rule. Spinning up the legal-advisor to read one sentence. The setup cost (system prompt, tool definitions, context window) dwarfs the actual work. It's hiring a consultant to check whether your front door is locked.
The agent's setup cost (in tokens) is more than 2× the work itself.
ANTI 04"Throw 10 agents at it" sprawl
The shape. Enterprise deployments now average around 12 agents per workflow, and roughly half operate in isolation — firing off, returning output nobody synthesizes, and burning tokens in the background. Gartner projects more than 40% of agentic AI projects will be cancelled by 2027, largely because the costs outran the demonstrable value. ND users with high-energy enthusiasm are especially prone to this — designing the org chart is more dopaminergic than running it.
You can name your agents but you can't name what each one's last output changed.
ANTI 05AutoGPT-style infinite loops
The shape. An agent given a goal but no explicit termination condition. It loops — plan, act, observe, plan, act, observe — chasing a vague success criterion forever, racking up tokens until you notice the bill. Every loop needs three explicit exits: a success state, a failure state, and a max-iterations cap. Missing any one of them turns an agent into a money pump.
You can't articulate, in one sentence, the condition that makes the agent stop.
ANTI 06Telephone-game pipelines
The shape. Five or more sequential handoffs where each agent summarizes the previous output for the next. By step five the nuance is gone, the qualifiers are gone, the specific examples are gone, and the final agent is working from a sanitized blur. A single well-prompted agent reading the original source outperforms the entire chain.
When you read the final output, it doesn't reference any specific from the original input.

When single-agent wins

🎯
Five conditions where one agent reliably beats a council
These are not edge cases. For an estimated 70% of real workloads, one of these conditions applies — which is why a single well-prompted agent delivers 90-95% of multi-agent outcomes at 30-40% of the cost.
01
Sequential reasoning tasksStanford research finds that single agents with adequate thinking budget match or outperform multi-agent setups on complex reasoning. When the work is "think hard about one thing," one chain-of-thought beats five summaries of five chains-of-thought.
02
Exploratory, undefined problemsYou can only decompose what you understand. If the subtasks aren't yet clear, multi-agent has nothing to dispatch. Explore with one agent first, decompose later if the problem turns out to need it.
03
Tasks under ~30 minutes of single-agent workCoordination overhead on a multi-agent setup is roughly fixed. On a short task, the overhead consumes the time saved — you end up slower than just doing the task with one agent.
04
When auditability mattersMulti-agent introduces opacity — you can read each agent's output but not why the synthesis weighted them the way it did. One agent's chain-of-thought is fully traceable. For decisions you may need to defend later, traceability beats parallelism.
05
When you have no synthesis planA multi-agent system is only as good as the step that combines the outputs. If you don't know how you'll synthesize before you dispatch, you're going to get five answers and choose the first one that looks confident — which is worse than asking one good question.

The ND-specific decision rule

🧠
Match agent complexity to decision stakes, NOT to task excitement
ADHD hyperfocus is biased toward whatever feels engaging to design, not whatever delivers the best output. Elaborate multi-agent setups feel engaging to design. The correct check is two diagnostic questions:
  1. Does the output quality genuinely require this architecture? Would the result be measurably worse with one well-prompted agent?
  2. Would I recognize a good 3-agent output vs. a good single-agent output on this specific task? If you can't tell the difference, you don't need the complexity — you're paying for invisible work.
If either answer is "no" or "I'm not sure," the right move is a single agent with a sharper system prompt. The architecture excitement is a tell — not a signal.

The recap

Reach for a council when…

Disagreement catches blind spots. You need adversarial review of a decision where you can't trust your own perspective — that's pattern 2 (Devil's Advocate) or pattern 3 (Pre-Mortem Council).

Parallelism saves real time. You have four or more genuinely independent questions and decomposing them is cheaper than running them in sequence — that's pattern 4 (Parallel Research Dispatch).

Deep specialization genuinely differs. Two domains would frame the same question incompatibly, and you need both frames — that's pattern 1 (Specialist Consultation), and occasionally pattern 5 (Review Board).

Reach for a single agent for…

Everything else. With a strong system prompt, a clear output format, and a single named role, one agent handles the 70% of work that doesn't structurally need a council. The remaining 30% is where Track D earns its keep — and that's plenty.
📚
If you take one thing from this track
Anthropic's primary design principle — "prefer simplicity first" — is the most important sentence in the entire multi-agent literature. The pattern that wins most often is the pattern with the fewest moving parts. The skill being practiced in Track D is restraint, not architecture.
