When to build vs install

🛠
The decision rule
Search the marketplaces first. If a vetted skill or plugin covers 80% of what you need, install it and adapt with a small CLAUDE.md instruction rather than building from scratch. Build your own when (a) the work involves private domain knowledge, (b) no community option exists or the quality is too uneven to trust, or (c) the workflow is yours and yours alone and you want one source of truth on disk. Building is faster than people think for Patterns 1 and 2, and dramatically slower for Pattern 5 — budget accordingly.

Pattern 1: Simple Skill (~30 min)

01 Folder + SKILL.md with YAML frontmatter

No scriptingAuto-loads on description matchEasiest to share
The smallest unit of expertise. Claude reads the frontmatter description, decides whether to load the body, and applies the instructions when relevant. The whole "skill" is a Markdown file with a header — nothing else required.
---
name: notion-content-brief
description: Use when creating content for social posts, Substack, or template copy.
---
# Content Brief

## Brand Voice Rules
- Never use hollow affirmations
- Lead with a specific insight or data point
- Code blocks for any user-facing copy
- ND-aware framing: energy levels, not pressure

## Output Format
1. One-line hook
2. Three bullets of insight
3. Single concrete next step

Pattern 2: Skill with Scripts

02 SKILL.md + a scripts/ subdirectory

Local automationInherits shell permissionsScope with allowed-tools
When the skill needs to actually run something — build a report, scrape a file, format an output — you drop scripts into a scripts/ subdirectory and reference them from the body of SKILL.md. Scripts run with Claude Code's Bash permissions, which is the security caveat.
my-skill/
├── SKILL.md           # references scripts/ by relative path
└── scripts/
    ├── fetch_data.py
    └── format_report.sh
⚠
Security caveat
Scripts execute with whatever permissions your shell has. When sharing the skill, scope it with allowed-tools in the frontmatter so collaborators don't accidentally hand it write access it doesn't need.

Pattern 3: Custom Subagent

03 A file in ~/.claude/agents/

Isolated contextCheaper model routingSpawned via Task tool
A subagent runs in its own context window with its own system prompt and tools list. Useful for long-running specialist tasks where you don't want to pollute the main session. Routing to Haiku for routine work cuts costs significantly.
---
name: template-qa-agent
description: Runs QA on Notion templates before publishing.
model: claude-haiku-4-5
allowed-tools: Bash, Read, WebFetch
---
You are a quality assurance specialist for Notion templates.

For each template:
1. Verify all internal links resolve
2. Check that callouts are color-consistent
3. Validate that view filters return expected rows
4. Flag any AI-tells in the user-facing copy
5. Return a structured pass/fail report with line-level findings

For inspiration, the VoltAgent/awesome-claude-code-subagents repo has 100+ specialized agents you can copy and adapt — reviewers, planners, debuggers, domain specialists.

Pattern 4: Multi-Skill Plugin

04 Plugin directory with skills, agents, and an MCP

Distribution formatBundle anythingPublishable to marketplace
Once you have multiple skills that work together — plus maybe a custom agent and an MCP server — a plugin is the distribution wrapper. The structure is opinionated:
my-plugin/
├── .claude-plugin/
│   └── plugin.json          # name, version, components
├── skills/
│   ├── content-brief/SKILL.md
│   └── template-qa/SKILL.md
├── agents/
│   └── reviewer.md
└── mcp/
    └── server.js            # optional inline MCP server

Publish by opening a PR to anthropics/claude-plugins-official, or self-host as a Git URL plugin source for private use.

Pattern 5: Standalone Agent (Agent SDK)

05 Headless Claude Code loop in Python or TS

Production automationCron / monitor / pipelineHighest setup cost
When you need an agent running on a schedule, in a CI pipeline, or as a background process, the Agent SDK gives you the same agent loop Claude Code itself uses — minus the interactive terminal. Outputs stream, tools work, you handle the orchestration.
import anthropic

client = anthropic.Anthropic()

with client.beta.claude_code.sessions.stream(
    model="claude-opus-4-7",
    system_prompt="You are a market research agent. Each morning you fetch the top "
                  "competitor pricing changes, score them, and update the Notion DB.",
    tools=[
        {"type": "mcp", "server": "notion"},
        {"type": "mcp", "server": "exa"},
    ],
) as stream:
    for event in stream:
        print(event)

This is the same loop the Claude Code CLI runs — you drive it programmatically instead. Schedule with launchd, cron, or APScheduler depending on platform.

Anthropic's official framing
"Skills add expertise any agent can apply; subagents run independently with context isolation." If you internalize that one line you'll pick correctly nearly every time.

Hybrid Local + Cloud

The other half of "build your own" is deciding which model runs each piece. By mid-2026, the gap between local 7–14B models and cloud Sonnet has narrowed enough that routing the right work to a local Ollama instance is the single largest cost lever most heavy users have.

The 70/30 economic argument

Roughly 70–80% of production LLM queries don't actually need a frontier cloud model — classification, short-answer Q&A, structured extraction, code completion, summarization. A well-routed hybrid offloads that majority to a local model at near-zero marginal cost and reserves cloud spend for the work that genuinely needs Sonnet or Opus.

Wire Ollama into Claude Code

Two MCP servers connect Claude Code to a local Ollama runtime. Pick one based on what you want to do.

Option A: OllamaClaude (recommended for coding tasks)

String-based tools (generate, explain, review, refactor) plus file-aware variants that read files server-side. The vendor claims up to a 98.75% token reduction on file-heavy workflows, because Claude sends a path instead of the file contents.

# Clone and install
git clone https://github.com/Jadael/OllamaClaude
cd OllamaClaude
npm install

# Pull a starter local model
ollama pull qwen3:8b

Then add it to your Claude Code MCP config (~/.claude.json):

{
  "mcpServers": {
    "ollama": {
      "command": "node",
      "args": ["C:/full/path/to/OllamaClaude/index.js"]
    }
  }
}

Option B: rawveg/ollama-mcp

Exposes the full Ollama SDK as 14 MCP tools — pull models, list models, generate, embed, stream, manage. Better when you're building pipelines that need to manage Ollama itself, not only call it.

# OpenAI-compatible API is already running at localhost:11434
# No API key needed for local use
curl http://localhost:11434/api/tags

The Ollama API is OpenAI-compatible, so if you're writing your own agent code you can swap base_url in the OpenAI SDK and aim it at localhost:

from openai import OpenAI

local = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
response = local.chat.completions.create(
    model="qwen3:8b",
    messages=[{"role": "user", "content": "Summarize this in 3 bullets..."}],
)

When NOT to bother

🤭
Honest caveats

Local-cloud hybrid is genuinely worth it for the right user. It's a waste of time for the wrong one. Skip the setup when:

  • You run fewer than ~50 messages a day. The savings won't cover the hardware time and maintenance attention.
  • You don't have spare hardware. Running a local model on the same machine you're working on burns power and adds latency to everything else. A dedicated Mac Mini or workstation changes the calculus.
  • Your work is reasoning-heavy. Architecture decisions, deep synthesis, complex agentic tasks — Qwen3 8B's HumanEval score is 76 versus Sonnet's high-80s-to-90s. The capability ceiling is real.
  • You'd be swapping for a nano-tier model anyway. Cloud nano tiers (~$0.20 per million tokens) almost never break even against the hardware-time cost of running locally.
  • You haven't audited which tasks would actually route local. Build the cost model first. Many people assume their cloud spend is wasteful and discover most of it was already going to genuinely Sonnet-worthy work.

Privacy and offline capability are the strongest non-cost arguments. If GDPR or HIPAA structure matters to your workflow, or you spend real time on airplanes and hotel wifi, local is worth setting up even when the dollar math is close.
