Track D Councils & Multi-Agent

What this track teaches

🧠
The premise
Multi-agent is not better. It is different — and most of the time it costs 3-5× more tokens for outcomes a single well-prompted agent could match. This track gives you the vocabulary to talk about councils precisely, the six patterns ND professionals can actually use, and the anti-patterns that quietly burn money and attention. We end where Anthropic begins: prefer simplicity first.

The precise vocabulary

People use these words interchangeably. They aren't. Mixing them up is how you end up with a "council" that's really a pipeline, or a "subagent" that's really a long prompt. Get the labels right and the rest of the track lands cleanly.

TermWhat it actually means
AgentAn LLM running an agentic loop: plan → act → observe → plan again. The loop is what distinguishes an agent from a chatbot.
SubagentAn agent spawned by another agent. It runs in a separate Claude conversation with its own system prompt, its own tool allowlist, and a fresh context window. It does not share the parent's context.
Council / PanelMultiple agents or personas reviewing one problem from different angles. Maps to Anthropic's orchestrator-workers and agent-teams patterns.
War RoomParallel multi-expert analysis: 3-5 agents running simultaneously, each from a distinct domain.
Review BoardSequential: A drafts, B critiques, C synthesizes. This is the generator-verifier pattern with a synthesis step on top.
PipelineAn assembly line where one output feeds the next input. This is a workflow, not multi-agent. It suffers telephone-game degradation.
Agentic LoopThe plan-act-observe cycle inside any agent. A loop without an explicit termination condition (success, failure, or max iterations) is an anti-pattern, not an agent.
⚠️
"Pipeline" is the most misused word
If your "multi-agent system" is really five steps where each one waits for the previous one, you have a workflow. That's fine. But labeling it "multi-agent" buys you the cost overhead without the speedup — because there is no speedup when the work is sequential.

Anthropic's foundational position

The clearest writing on this comes from Anthropic's own engineering team. Their Building Effective Agents (December 2024) is the canonical reference. The primary design principle is short enough to tape to your monitor.

"Prefer simplicity first. When a workflow with a single agent works, use it. Add complexity only when the task structurally requires it." — Anthropic, Building Effective Agents
📚
The three structural conditions that justify multi-agent
Anthropic names three. If your task does not fit one of these, a single agent is the better tool:
  1. Tasks too long for a single context window — research over thousands of pages, codebases that won't fit, document collections that exceed even Claude's million-token window.
  2. Tasks that benefit from independent agents checking each other's work — when verification by a fresh perspective genuinely catches errors a single chain-of-thought misses.
  3. Parallel independent workstreams — multiple questions that don't depend on each other and would take N times longer in sequence.
Anthropic's own production research system uses orchestrator-worker for condition #3 — simultaneous independent investigation of distinct subtopics.

The 5 official coordination patterns

From Anthropic's multi-agent coordination patterns guide. These are the named structures — every multi-agent design you'll see in the wild is a version of one of these.

PatternUse whenDon't use when
Generator-VerifierQuality is critical and the verification criteria are explicit and inspectableVerification criteria are vague or subjective — verifier just becomes a yes-machine
Orchestrator-SubagentStart here. Widest range of tasks, least coordination overhead. Anthropic's default recommendationSubagents need to share intermediate findings with each other mid-task
Agent TeamsSustained multi-step work that genuinely needs deep, separate domain expertiseAgents must share real-time discoveries — teams can't gossip mid-flight
Message BusEvent-driven workloads — security alerts, triage queues, asynchronous handoffsYou need transparent, debuggable, linear behavior
Shared StateGenuinely collaborative findings build cumulatively in one placeYou don't have a robust termination condition — shared state without an exit is a stuck loop

The real cost numbers

This is the part most "multi-agent is the future" posts skip. The numbers are from production data and they are not subtle.

  • Subagent-heavy workflows cost 200-500% more tokens than the same work done by a single agent.
  • ~20,000 tokens of context-loading overhead per subagent before any actual work happens — that's the system prompt, tool definitions, and task framing every spawn pays for.
  • Agent teams in plan mode use roughly 7× more tokens than a standard single-agent session.
  • For an estimated 70% of workloads, single agents deliver 90-95% of multi-agent outcomes at 30-40% of the cost.
💰
The one cost lever worth knowing
The CLAUDE_CODE_SUBAGENT_MODEL environment variable lets you run the orchestrator on Opus and subagents on Sonnet or Haiku. This is the single highest-leverage cost reduction you can make without changing your architecture — most subagent work is mechanical retrieval and summarization that doesn't need Opus-level reasoning.

Why this track lives last

Every other track in this MasterClass — multi-AI routing, prompting, RAG — teaches you to get more out of a single agent. This one teaches you when to add a second. The order is deliberate.

🎯
The ND-specific trap
Designing a five-agent council is more dopamine-engaging than writing one good system prompt. ADHD brains reach for elaborate architectures because the architecture itself is the interesting puzzle. The output rarely justifies it. Master the single-agent skills in Tracks A-C first — you'll need them anyway, and you'll know when a council is genuinely earning its keep.
